# OpenClaw Manager 产品叙事

## 为什么做这个

OpenClaw 做对的一点非常重要：它把 agent 放进了人已经会用、已经愿意打开的聊天窗口里。这个动作的价值极大，因为它让 agent 第一次以非常低的心智成本进入了普通人的日常。一个人不需要先打开电脑坐直腰板才能开始体验 agent；他只需要像发消息一样说一句话。这是 OpenClaw 最正确、也最有传播力的一步。它抓住的是"简单体验"，也就是最容易可用的体验。

但也正因为它借用了聊天这个入口，它几乎注定会继承聊天范式的原罪。聊天天然是线性的，天然缺乏结构化状态。于是 agent 一旦从"玩一玩""查个东西"进入"真的干活"，问题就会全部暴露出来：对话会越来越长，真正重要的目标和中间决定会被淹没；输出会不稳定，因为每一次尝试都只是漂浮在上下文里，没有被稳固成可以回退、可以恢复、可以比较的执行对象；生产过程不可观测，因为只能看到消息而不是状态或任务。对于一类频繁进行多线程脑力劳动的人，比如研究者、开发者，这都是一个需要长期维护复杂认知任务的负担。我认为这是致命的。

## 核心定位

这个项目的核心，不是"做一个更好的聊天界面"，而是把 OpenClaw 从"消息流里的 agent"提升成"可管理的任务系统"。我最坚持的就是：真正高价值的 agent，不应该只存在于对话里，而应该存在于 session、状态、尝试、证据这样的生产环节中。这个管理工具，是期望可以把 OpenClaw 从一个低门槛入口，托举成一个可以承载严肃工作流的控制面。

## 产品形态

这是一个 OpenClaw-native 的 agent 管理层，一个位于聊天之上、但不破坏聊天入口的最小控制平面。它的 MVP 不是平台或新客户端，而是一个让 OpenClaw 用户第一次获得多 session 管理、执行可视化、尝试可回退、状态可恢复、skills 可组织的轻量操作层。这个 MVP 力图把原本散在聊天流中的长期任务收编成 session，把每次执行沉淀成 run，把事件、阻塞、人工决策、checkpoint、summary 固化下来，让用户不再只能"翻聊天记录"，而是能真正管理任务。它让一个人第一次能够同时拥有 OpenClaw 的统一入口、长记忆和 skills 生态，又不再被线性聊天绑架。

## 目标用户与场景

它服务的是"用过 IDE 或者 Claude Code，现在对 OpenClaw 不那么满意的人"。这些人最典型的场景，不是一次性问答，而是跨时间、跨尝试、跨信息源、跨注意力的持续任务。在这些场景里，用户真正需要的不是更花哨的回答，而是三种稀缺能力：

1. **多 Session 能力**：把不同任务从单一对话流里解放出来，形成可并行、可切换、可排序的任务线程。
2. **可观测性**：不仅知道 agent 回答了什么，还知道它做过哪些尝试、卡在什么地方、为什么停、哪一步需要人类介入。
3. **恢复与回退**：当输出不好、上下文漂移、任务中断时，用户可以回到一个可信状态重新开始，而不是重新打一轮运气。

## 更深层的命题

更深一层看，这个项目的真正价值还不只是"让 OpenClaw 更顺手"。我在试图解决一个更大的命题：当 agent 开始成为新的生产参与者时，人和 agent 之间到底需要什么样的中介层。随着 agent 能力的发展，它理解人类意图的能力会越来越强。但这个中间层并不只是"怎么把用户放到 agent 面前"的问题，"怎么让 agent 可以直接向人类显现自己"也同样重要。

单人公司、研究型个人生产、超小团队协作，最缺的恰恰不是更多能力或注意力，而是把注意力组织起来的方式。利用一个简单的 skill，就可以重现一种新生产方式里的最小管理基础设施：它让单个人能够同时调度多个 agent 线程、保存过程、回收经验、复用 skill，同时又可以只维持统一入口和低心智负担。
